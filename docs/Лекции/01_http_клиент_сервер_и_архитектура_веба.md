# Лекция 01. Клиент-серверная архитектура, HTTP и устройство веба

## Аннотация

Лекция посвящена базовой архитектуре веб-приложений. Рассматривается модель клиент-сервер, устройство HTTP-запросов и ответов, методы HTTP, URL, заголовки, коды состояния, HTTPS и роль веб-сервера в современной многозвенной архитектуре.

- Увидеть, зачем в production используют nginx и сервер приложения.

## 1. Клиент-серверная архитектура

Классический веб работает по модели запрос-ответ.

Клиент отправляет запрос серверу. Сервер обрабатывает запрос и возвращает ответ. Постоянной связи в классическом HTTP нет: каждое действие начинается с отдельного запроса.

В вебе клиентом чаще всего выступает браузер, а сервером — программа, которая принимает HTTP-запросы, выполняет бизнес-логику и возвращает данные.

Преимущества клиент-серверной модели:

- несколько пользователей работают с общими данными;
- данные централизованно хранятся на сервере;
- клиенту не нужно иметь всю базу данных локально;
- можно управлять доступом, безопасностью и бизнес-правилами на сервере.

Недостатки:

- нужно разрабатывать две части: клиент и сервер;
- появляется сеть со своими задержками и ошибками;
- нужно думать о безопасности;
- клиент не всегда имеет все данные сразу;
- приходится проектировать протокол взаимодействия.

## 2. Тонкий и толстый клиент

Тонкий клиент почти не содержит бизнес-логики. Он в основном отображает данные, полученные от сервера.

Толстый клиент содержит часть бизнес-логики и может выполнять сложные операции на стороне пользователя.

Изначально браузер задумывался как тонкий клиент: сервер отдает HTML, браузер отображает. С появлением JavaScript браузер стал способен выполнять сложную логику, поэтому современные веб-приложения могут быть толстыми клиентами.

Пример толстого клиента: Figma, сложные редакторы, игры и интерактивные приложения в браузере.

Правило безопасности: клиенту нельзя доверять. Все критические проверки, права доступа и валидация должны выполняться на сервере.

## 3. HTTP

HTTP расшифровывается как HyperText Transfer Protocol, то есть протокол передачи гипертекста.

Изначально HTTP использовался для передачи HTML-документов. Сейчас через HTTP передаются почти любые данные: HTML, CSS, JavaScript, изображения, файлы, JSON, сообщения API.

HTTP/1.1 является текстовым протоколом. Это значит, что запрос и ответ можно прочитать как обычный текст.

Пример простого запроса:

```http
GET / HTTP/1.1
Host: example.com

```

Пример ответа:

```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 1256

<!doctype html>
<html>...</html>
```

## 4. Структура HTTP-сообщения

HTTP-сообщение состоит из трех частей:

- стартовая строка;
- заголовки;
- тело сообщения.

Заголовки отделяются от тела пустой строкой.

В запросе стартовая строка содержит метод, путь и версию протокола:

```http
GET /catalog/?page=2 HTTP/1.1
```

В ответе стартовая строка содержит версию протокола, код статуса и текстовое описание:

```http
HTTP/1.1 404 Not Found
```

## 5. URL

URL задает адрес ресурса в сети.

Общая структура:

```text
scheme://user:password@host:port/path?query#fragment
```

Основные части:

- `scheme`: протокол, например `http`, `https`, `ssh`.
- `user:password`: данные пользователя, сейчас используются редко.
- `host`: доменное имя или IP-адрес.
- `port`: порт сервера, например `80`, `443`, `8000`.
- `path`: путь до ресурса на сервере.
- `query`: GET-параметры после `?`.
- `fragment`: якорь после `#`, обрабатывается браузером и не отправляется на сервер.

Пример:

```text
https://example.com/catalog/phones/?brand=samsung&page=2#reviews
```

## 6. Заголовок Host

Начиная с HTTP/1.1 заголовок `Host` обязателен.

Причина: сетевое соединение устанавливается по IP-адресу, но на одном IP может находиться несколько сайтов. Заголовок `Host` сообщает веб-серверу, какой именно сайт запрашивает клиент.

Пример:

```http
GET / HTTP/1.1
Host: example.com
```

## 7. HTTP-методы

### GET

`GET` используется для получения данных.

Особенности:

- не должен менять состояние сервера;
- может кешироваться браузером и прокси;
- параметры передаются в URL;
- не подходит для передачи паролей, токенов и других чувствительных данных.

Плохой пример:

```text
/user/set-password/?password=123456
```

Так делать нельзя: пароль попадет в историю браузера, логи, рефереры и может быть проиндексирован.

### HEAD

`HEAD` похож на `GET`, но возвращает только заголовки без тела ответа.

Используется для проверки существования ресурса, размера файла, даты изменения и кеширования.

### POST

`POST` используется для отправки данных на сервер и может менять состояние сервера.

Особенности:

- данные передаются в теле запроса;
- запрос не считается идемпотентным;
- повторная отправка может создать повторное действие;
- не кешируется как обычный `GET`.

Пример: отправка формы, создание комментария, оформление заказа.

Если пользователь дважды отправит одинаковый `POST` на создание комментария, сервер может создать два комментария.

### PUT, PATCH, DELETE

Эти методы особенно важны при проектировании REST API.

- `PUT`: полная замена ресурса.
- `PATCH`: частичное изменение ресурса.
- `DELETE`: удаление ресурса.

В старых веб-приложениях часто использовали только `GET` и `POST`, но современные API обычно используют весь набор методов.

## 8. HTTP-заголовки

Заголовки передают служебную информацию о запросе или ответе.

Примеры:

- `Host`: имя сайта.
- `Content-Type`: тип тела сообщения.
- `Content-Length`: размер тела.
- `User-Agent`: информация о клиенте.
- `Referer`: откуда пользователь перешел.
- `Accept-Encoding`: поддерживаемые способы сжатия.

Важно: большинству заголовков нельзя доверять. Клиент может указать произвольный `User-Agent`, `Referer` и другие значения.

## 9. Коды состояния HTTP

Коды состояния делятся на группы.

### 1xx: информационные

Сообщают о промежуточном состоянии, например переключении протокола.

### 2xx: успех

Запрос успешно обработан.

Частые примеры:

- `200 OK`: успешный ответ;
- `201 Created`: ресурс создан.

### 3xx: перенаправления

Сервер просит клиента перейти по другому адресу.

Важные примеры:

- `301 Moved Permanently`: постоянный редирект;
- `302 Found`: временный редирект.

С `301` нужно быть осторожным: браузер может запомнить его надолго. Ошибка в permanent redirect может привести к тому, что пользователь будет постоянно перенаправляться не туда.

### 4xx: ошибка клиента

Проблема на стороне запроса пользователя.

Примеры:

- `400 Bad Request`: некорректный запрос;
- `401 Unauthorized`: требуется авторизация;
- `403 Forbidden`: нет прав доступа;
- `404 Not Found`: ресурс не найден;
- `418 I'm a teapot`: шуточный статус.

Важно возвращать правильный код. Если страницы нет, нужно вернуть `404`, а не `200 OK` со страницей, где написано «404».

### 5xx: ошибка сервера

Проблема на стороне сервера.

Примеры:

- `500 Internal Server Error`: внутренняя ошибка приложения;
- `502 Bad Gateway`: промежуточный сервер не смог получить корректный ответ от backend-сервера.

В production нельзя показывать пользователю stack trace, SQL-запросы, настройки и другую отладочную информацию.

## 10. HTTP и HTTPS

HTTP передает данные в открытом виде. Любой участник сети, способный перехватить трафик, может увидеть запросы и ответы, а иногда и изменить их.

HTTPS — это HTTP поверх защищенного TLS/SSL-канала.

HTTPS обеспечивает:

- шифрование данных;
- защиту от чтения трафика посредником;
- защиту от модификации ответа;
- проверку подлинности сервера через сертификат.

В HTTPS используются сертификаты, асимметричная криптография и алгоритмы согласования общего секрета, например Diffie-Hellman.

Практический вывод: любой современный публичный сайт должен работать по HTTPS.

## 11. Веб-сервер и сервер приложения

Простейший TCP/HTTP-сервер может принять соединение, прочитать запрос и отправить ответ. Но у такого сервера сразу возникают проблемы:

- нужно обслуживать много клиентов одновременно;
- некоторые клиенты имеют медленное соединение;
- нужно эффективно отдавать статические файлы;
- нужно управлять соединениями и таймаутами;
- нужно экономить память и CPU.

Поэтому современные веб-приложения обычно имеют многозвенную архитектуру.

Типовая схема:

```text
Браузер -> nginx -> сервер приложения -> база данных
```

nginx берет на себя:

- прием HTTP-соединений;
- работу с большим числом клиентов;
- отдачу статических файлов;
- TLS/HTTPS;
- проксирование запросов к серверу приложения.

Сервер приложения занимается бизнес-логикой: Django, авторизация, работа с данными, формирование ответа.

## Термины

- Клиент: программа, отправляющая запросы серверу.
- Сервер: программа, принимающая запросы и возвращающая ответы.
- HTTP: протокол передачи данных в вебе.
- HTTPS: защищенный HTTP поверх TLS.
- URL: адрес ресурса.
- Заголовок: служебная информация HTTP-сообщения.
- Тело запроса: данные, передаваемые в запросе или ответе.
- Редирект: перенаправление клиента на другой адрес.
- nginx: популярный веб-сервер и reverse proxy.
