# Лекция 00. Введение в курс и исторические основы веб-разработки

## Аннотация

Лекция открывает курс по веб-разработке и задает общий контекст: организацию обучения, требования к итоговому проекту, необходимый инструментарий и исторические причины появления современных веб-технологий. Главная идея занятия: современный веб возник как ответ на проблемы переносимости, совместимости и взаимодействия программных систем.

## Цели занятия

- Понять структуру курса и ожидаемый результат обучения.
- Разобрать требования к итоговому fullstack-проекту.
- Определить базовые инструменты разработки.
- Проследить путь от первых ЭВМ к открытым системам и вебу.
- Осознать роль стандартов, переносимости, масштабируемости и API.

## План

1. Организация курса и формат работы.

## 1. Организация курса

Курс посвящен fullstack веб-разработке. В ходе курса будут рассмотрены:

- базовые принципы работы веба;
- HTTP, HTML, CSS и клиент-серверная архитектура;
- backend-разработка на Django;
- frontend-разработка на Vue;
- REST-взаимодействие между клиентом и сервером;
- элементы архитектуры программного обеспечения;
- основы администрирования и развертывания проектов.

Материалы курса публикуются в Moodle и в GitHub-репозитории. Записи занятий и дополнительные материалы также используются как часть учебного процесса.

## 2. Итоговый проект

В течение курса каждый студент разрабатывает собственный веб-проект. В конце курса проект защищается.

Минимальные требования:

- Django-проект с настроенной административной панелью.
- Не менее двух страниц, отрисованных средствами Django-шаблонов.
- Основная интерактивная часть на Vue.
- Асинхронное взаимодействие frontend и backend через REST API.
- Желательно использование WebSocket для знакомства с real-time-взаимодействием.
- Исходный код размещен на GitHub.
- Желательно развертывание на VPS или другом публичном сервере.

Формат защиты:

- 3-5 минут на доклад.
- Краткое описание идеи проекта.
- Объяснение прикладной задачи и целевой аудитории.
- Описание архитектурных решений.
- Демонстрация работающего проекта.
- Ссылка на исходный код.

## 3. Инструменты

### Среда разработки

Можно использовать любой удобный инструмент:

- Visual Studio Code как популярный редактор с системой расширений.
- PyCharm как полноценную IDE для Python и Django.
- Visual Studio, если студент привык к этой среде.
- Neovim или другие редакторы при наличии опыта.

Важно не столько выбрать конкретную IDE, сколько настроить подсветку, автодополнение, поддержку Python, Django, JavaScript и Vue.

### Backend

Основной стек backend-части:

- Python версии не ниже 3.12.
- Django 6 или актуальная поддерживаемая версия Django.
- `uv` для управления зависимостями и виртуальными окружениями.

`uv` используется как современная альтернатива ручной работе с `pip`, `venv` и `requirements.txt`. Он позволяет управлять зависимостями проекта, создавать окружение и запускать команды внутри него.

### Frontend

Основной стек frontend-части:

- Node.js версии не ниже 20.
- `npm` как пакетный менеджер.
- Vue 3.
- Vite для сборки frontend-приложения.

### Git и Unix-среда

Git необходим для хранения исходного кода и публикации проекта на GitHub.

Желательно иметь доступ к Unix-подобной среде:

- Linux;
- macOS;
- WSL на Windows.

Это важно, потому что серверное окружение, Docker, Redis и многие инструменты backend-разработки чаще всего работают в Linux-среде.

## 4. Как выбрать идею проекта

Хороший проект должен решать прикладную задачу, а не быть абстрактной демонстрацией технологий.

Варианты поиска идеи:

- Собственная проблема: автоматизация учебы, работы, хранения заметок, фотографий, задач.
- Проблема знакомых или потенциальных пользователей.
- Аналог существующего сервиса, чтобы разобраться в его устройстве.
- Типовой учебный проект: интернет-магазин, таск-трекер, блог, социальная сеть, сервис комментариев.

Главное требование: пользователь должен активно взаимодействовать с системой. Простая статическая страница не подходит. В проекте должны быть данные, действия пользователя, изменение состояния, желательно взаимодействие нескольких пользователей.

## 5. Исторический контекст

### Первые ЭВМ и проблема переносимости

Первые электронные вычислительные машины были крупными, дорогими и использовались в основном для математических расчетов. Программы писались под конкретную архитектуру, поэтому перенос кода между машинами был сложным или невозможным.

Проблема: если менялась машина, приходилось переписывать программу под новый набор инструкций.

Решение: языки высокого уровня и компиляторы.

Пример: Fortran позволил писать код на более абстрактном языке и компилировать его под разные машины. Это стало первым важным шагом к переносимости ПО.

### Unix и язык C

Unix появился как простая, переносимая и удачно спроектированная операционная система. Переписывание Unix на C сделало систему переносимой между разными аппаратными платформами.

Ключевая идея: если есть компилятор C для новой архитектуры, можно перенести на нее значительную часть системного ПО.

### Персональные компьютеры

Apple I и IBM PC сделали компьютеры массовым продуктом. Особенно важной стала открытая архитектура IBM PC: разные производители могли выпускать совместимые компоненты.

Это привело к снижению цен, росту рынка и массовому распространению персональных компьютеров.

### GNU, Linux и открытый софт

Проект GNU и затем Linux заложили основу современной культуры открытого ПО. Linux стал ядром операционной системы, а инструменты GNU дополнили его до полноценной Unix-подобной среды.

Важно различать:

- открытый исходный код, когда код доступен для изучения и модификации;
- открытые системы, когда система построена по стандартам и может взаимодействовать с другими системами.

## 6. Открытые системы и стандарты

Современная разработка невозможна без стандартов. Программы, протоколы и форматы должны быть совместимы.

Открытая система строится так, чтобы:

- взаимодействовать с другими системами;
- использовать стандартные протоколы и форматы;
- не требовать переписывания всего с нуля при изменении окружения;
- быть переносимой и расширяемой.

Примеры стандартных решений:

- HTTP вместо собственного протокола передачи данных.
- PostgreSQL или SQLite вместо самописного хранилища.
- JSON вместо собственного формата обмена.
- Django вместо полностью самописного backend-фреймворка.
- API вместо прямого доступа к внутренней базе данных чужой системы.

## 7. API как граница между системами

API задает контракт взаимодействия между системами. Это позволяет менять внутреннюю реализацию, не ломая внешнее взаимодействие.

Плохой подход: одна система напрямую читает таблицы базы данных другой системы.

Проблемы такого подхода:

- структура базы может измениться после обновления;
- можно нарушить целостность данных;
- сложно сопровождать и отлаживать;
- системы становятся слишком сильно связаны.

Хороший подход: системы взаимодействуют через документированный API.

Преимущества:

- внутренняя структура может меняться независимо;
- контракт обмена остается стабильным;
- ответственность за данные остается внутри системы-владельца;
- проще сопровождать и расширять решение.

## 8. Принципы проектирования веб-систем

### Не изобретать велосипед

Перед тем как писать собственное решение, нужно проверить, нет ли стандартного инструмента, библиотеки, протокола или готового API.

Задача разработчика не в том, чтобы переписать весь стек с нуля, а в том, чтобы решить прикладную проблему пользователя.

### Масштабируемость

Система должна допускать рост нагрузки. Если проект внезапно получает много пользователей, архитектура должна позволять наращивать ресурсы: добавлять серверы, использовать балансировщик, разделять компоненты.

### Расширяемость

Требования меняются. Архитектура должна позволять добавлять новые функции без полного переписывания проекта.

### Переносимость

Хороший проект не должен быть жестко привязан к одной машине, одному окружению или одному поставщику.

### Понятный интерфейс

Пользовательский интерфейс должен учитывать привычки пользователей. Иногда привычное решение важнее красивого, но непонятного.

Примеры устоявшихся интерфейсных соглашений:

- самолетик означает отправку сообщения;
- три горизонтальные линии означают меню;
- корзина означает удаление;
- шестеренка означает настройки.

## Термины

- Fullstack-разработка: разработка backend и frontend частей приложения.
- Backend: серверная часть приложения.
- Frontend: клиентская часть приложения.
- API: программный интерфейс для взаимодействия систем.
- Открытая система: система, построенная по стандартам и способная взаимодействовать с другими системами.
- Open source: программное обеспечение с открытым исходным кодом.
- Масштабируемость: способность системы выдерживать рост нагрузки.
- Расширяемость: способность системы развиваться и добавлять новую функциональность.
- Legacy: старый код или система, которые нельзя просто выбросить, но сложно сопровождать.
